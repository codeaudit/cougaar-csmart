ABC Society Configuration Help

The ABC Society uses abstract domain plugins to generate an arbitrarily large and complex society, which behaves like a logistics supply chain.

The Society is organized by Communities, which pass Tasks between them. Each community consists of four Agents: Customer1, Customer2, Provider1 and Provider2.

Customer 1 and Customer 2 Agents periodically inject new demand Tasks into the system. Provider 1 and Provider 2 Agents supplies local assets for these tasks. When tasks are submitted, Provider1 attempts to satisify them with some local asset capacity. If that Agent cannot fulfill the requests, they are then allocated to the Provider2 Agent within that community.

If the Provider2 cannot fulfill the task locally, it allocates the task to the Provider2 Agent in another Community, depending on how the supply relationships have been established.

This society works in a purely execution mode: all tasks are matched against assets, and responses are always given with high confidence eventually. In particular, Tasks include deadlines for completion, and often requests will fail due to an exceeded deadline.

To build an ABC Society, you can specify parameters at several different levels.
Note: that all parameters have defaults, so that modifications at all levels are not required.

Parameters

You may edit parameters at the following levels:
  • Society Level
    Overall control parameters.
  • Community Level
    Overall community control parameters.
  • Agent Level
    Overall Agent control parameters.
  • Allocation Level
    Overall allocation control parameters.
  • Rule Level
    Overall Allocation Rule control parameters.
  • Tasks Level
    Overall control of root tasks (per agent).
  • Local Asset Level
    Overall control of local assets (per agent).

Society Level

 The Society level allows control over various aspects of the entire society.

Component Fields

Name Type Range
Community Count Integer Level Count - MAX
Level Count Integer 1 - MAX
Start Time (relative) Long 0 - MAX
Stop Time (relative) Long 0 - MAX
Task Verb String Array {Verb [, verb...]}


An ABC Society consists of one or more communities which in turn contain 4 agents (2 Customer and 2 Provider Agents). A society attempts to maintain a basic hourglass topology of communities by dividing the communities among levels. By default, there are 5 communities and three levels.
This results in the following topology:

Community 1 Community 2
Community 3
Community 4 Community 5

Changing the Community Count or changing the Level Count will regenerate a new community with an attempt to preserve the hourglass topology.

Note: There must always be at least as many communities as levels.

There are two types of agents within a society. Customer agents and Provider agents. Customer agents produce tasks and provider agents contain the assets to satisfy the task request. Tasks are submitted between Start Time and Stop Time. Tasks start getting generated at Start Time + current time and continue until Stop Time + current time.

Community Count

 Community Count specifies the total number of communities within the society.

Level Count

 Level Count Specifies the total number of levels within the society. The ABC Society attempts to maintain an hour glass topology where there are fewer communities at the inner levels than there are at the ends.

Start Time

 Start Time specifies the relative starting time in milliseconds. When the current time equals the start time, Customer Agents start to submit tasks for completion.

Stop Time

 Stop Time specifies the relative stop time in milliseconds. When the current time equals the stop time, Customer Agents stop submitting tasks.
Note: Stop time does not indicate that all agent communications stop, Provider Agents will continue to satisify all outstanding tasks beyond stop time.

Task Verb

 Task Verb indicates the Known tasks within the society. Since the society can contain multiple task verbs, they must be specified as an array of strings. Note: See Known issues for more information about Task Verb

Community Level

 The community level specifies all parameters that are specific to communities.

Component Fields

Name Type Range
Demand Integer 0 - MAX
Latitude Float -90 - 90
Longitude Float -180 - 180
Production Integer 0 - MAX


Each community has a specific amount of production and demand associated with it. The demand is proportional to the number of tasks produced by the Customer agents within the community. The higher the demand, the more tasks produced. Production is proportional to the amount of inventory on hand in the community. Producer agents that contain a local inventory satisfy all Tasks until their inventory is depleted. Once the inventory is delpeted, the Task must be satisified elsewhere, until the local inventory has been re-supplied. Increasing the amount of Production in a community increases the local inventory levels, which will eliminate the need for tasks to be satisfied by other communities.
Each community also has a location associated with it. This location specified with Latitude and Longitude is the center of the community. All agent locations are based on the center of the community.

Demand

 Demand is used to generate the number of tasks produced by the Customer agents. The higher the demand, the more tasks that are produced.
Note: Demand is part of a formula, not an exact task count -- A Demand of 5 does not equal 5 tasks.

Latitude

 Latitude specifies the Latitude coordinate of the center of the community.

Longitude

 Longitude specifies the Longitude coordinate of the center of the community.

Production

 Production is used to generate the amount of inventory on hand within the community. The higher the production value, the more local assets within the community.
Note: Production is part of a formula, not an exact inventory level count

Agent Level

 The Agent level specifies all parameters that are specific to an agent.

Component Fields

Name Type Range
Direction Float 0 - 360
Distance Integer 0 - Max
Supplies String Array {Supply [,Supply..]}


All agents are located relative to the center of the community. This location is specified with the direction and distance parameters. All agents that contain local assets (Provider Agents) must also specify a list of the agents that this agents supplies assets.

Direction

 Direction specifies the direction, in degrees from North, of the agent from the center of the community.

Distance

 Distance specifies the distance, in meters of the agent from the center of the community.

Supplies

 Supplies specifies a list of all agents, fully qualified names, that this agent serves as a provider.
Note: This parameter is left empty for Customer agents since they do not contain local assets.

The Allocation Level

 The allocation level specifies all parameters that are specific to an allocation.

Component Fields

Name Type Range
Execute Time Long 0 - MAX
Number of Allocation Rules Long 0 - MAX
Publish Time Long 0 - MAX
Response Time Long 0 - MAX
Successful Allocation Float 0 - 1
Transport Time Long 0 - MAX


All agents have an allocation table. This table is used to allocate all the known tasks for this agent. The Allocator can trys to calculate reduced Task deadlines, based on estimated times. By asking downstream components to complete a Task within reduced times, the Allocator hopes to have sufficient time to send a response the original customer, or perhaps attempt another Allocation if this one fails.

Execute Time

 Estimated Execute Time specifies the estimated time, in milliseconds, that it will take to satisify the task once it is allocated to an actual asset.

Number of Allocation Rules

 Number of Allocation Rules specifies the amount of allocation rules within an allocation table. When a new rule is added to the allocation table (by incrementing the number) they are added as empty rules. The CSMART user is required to fill in the Roles and Task Verb for each new rule.

Publish Time

 Allocation Publish Time specifies the time, in milliseconds, that it takes the allocator to publish an allocation.

Response Time

 Response Publish Time specifies the time, in milliseconds, that it takes the allocator to publish a simple response.

Successful Allocation

 Allocation Confidence Required specifies the condidence level of received allocation results.
0 - Indicates that all allocatoin results will be considered confident.
1 - Indicates that only allocations that specify complete confidence will be accepted.

Transport Time

 Estimated Transport time specifies the time, in milliseconds, that it takes for the allocator to transport the allocation to the remote agent.

The Rule Level

 The Rule Level specifies all parameters that are specific to an allocation rule.

Component Fields

Name Type Range
Roles String Array {Role [,Role..]}
Task Verb String


All agents can have multiple allocation rules.
Each rule consists of a Task Verb to allocate and a list of the Roles required of an Asset to be able to allocate this Task verb to it (all listed rules must be present). Multiple rules for the same task verb are permissible. Later rules will usually be for less dependable, or increasingly remote Assets or Organizations.

Roles

 Roles specifies a list of Roles associated with this allocations Task verb.

Task Verb

 Task Verb specifies a specific Task within the allocation table. This task must be know throughout the society. Note: See Known Issues for more information about the Task Verb

The Tasks Level

 The Tasks level specifies all parameters that are specific to the task level:

Component Fields

Name Type Range
Number of Tasks Long 0 - MAX


The Number of Tasks specifies the number of tasks known to this agent. Changing this number increments the number of tasks within the task file. When new tasks are added, it is the responsiblity of the CSMART user to add a Task Verb to the newly created task. By default, the Task Verb is empty.

Number of Tasks

 Number of Tasks specifies the number of tasks known to a specific agent.

Task Children

 The Task level contains 1 or more children. Each child represents a specific task verb that agent should generate.
Note: Only Customer agents generate Tasks.

Each task consists of the following properties:

Component Fields

Name Type Range
Chaos Factor Integer 0 - MAX
Deadline Long 0 - MAX
Interval between Tasks Long 0 - MAX
Task Verb String
Vitalness Factor Float 0 - 1


When the customer agent is started, it starts to submit a task verb to be satisfied. Tasks are repeatedly submitted at a specified Interval from the start time until the stop time of the society. Some amount of variability can be applied to the Interval by increasing the Chaos Factor for the task. Each task must be completed by a specific time. This time is specified with Deadline. If the task is not completed by the deadline, the customer's overall happiness will decrease. The amount this value decreases for each unsuccessful task is controlled by the Vitalness Factor.

Chaos Factor

 Chaos Factor is the amount of variablilty in the task interval. The larger the Chaos Factor, the more variablilty in the interval.

Deadline

 Deadline is the amount of time, in milliseconds, after the task has be submitted by which it must be completed.

Interval

 The Interval is the amount of time, in milliseconds, between tasks.

Task Verb

 The Task Verb is the task to be satisified.

Vitalness

 The Vitalness indicates how vital completion of a task is to overall agent happiness. The higher the vitalness, the unhappier the agent becomes if the task fails.

The Local Asset Level

 The local asset level has the following properties:

Component Fields

Name Type Range
Asset Name String
Average Execute Time Long 0 - MAX
Inventory Decrement Percent Long 0 - 100
Inventory Decrement Deviation Long 0 - MAX
Roles String Array {Role [,Role..]}
Time Deviation Long 0 - MAX


Every provider agent contains a local asset. This asset is used to satisfy task requests. Every asset contains an inventory level. When the asset is used to satisfy a task request, the inventory level decreases. The decrement amount for each Task completion depends on the Decrement Amount parameter. The inventory level can also have a certain amount of chaos applied to it by changing the Inventory Deviation parameter (for example, stock loss due to fire or theft). This chaotic inventory perturbation is applied before every inventory request. Each local asset can satisfy specific roles.
Note: An asset must list all of the roles required of it by the Allocation Rules above for it to be used for a given task verb.

When an asset is used to satisfy a task, it takes a specific amount of time to complete the task request. By adjusting the Time Deviation value, the time taken to satisfy a Task will vary randomly.

Asset Name

 Asset Name is the name of the Local Asset. Asset Name is just an identifier and is not used within the system.

Average Execute Time

 Average Execute Time is the average amount of time it takes to perform a specific task. Average Execute Time also controls how rapibly the local stock is refilled after a request.

Inventory Decrement Percent

 Inventory Decrement Percent defines the decrement amount for each task completion. The larger the decrement amount, the quicker the inventory can be depleted.

Inventory Decrement Deviation

 Inventory Decrement Deviation causes the inventory decrement rate to vary randomly. The larger the value, the more variable the decrement rate. With an 'Inventory Deviation' of 0, the reported inventory level is the exactly calculable from the previous requests of that asset.

Roles

 Roles is a list of roles that this asset can satisfy.

Time Deviation

 Time Deviation causes the time taken to satisfy a task to vary randomly (Gaussian). The larger the value, the more variable the time.
